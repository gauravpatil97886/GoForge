## What Interviewers Actually Test

At the 8-15 LPA band, interviewers follow a predictable pattern. Here is what each round covers and the exact weightage.

```
Round Weightage Breakdown
─────────────────────────────────────────────────────
1. Go Language Fundamentals         25%
2. DSA / Problem Solving            30%
3. Concurrency (Basic)              20%
4. REST API / Backend Design        15%
5. Behavioral / Communication       10%
─────────────────────────────────────────────────────
```

### Area 1 — Go Language Fundamentals (25%)

**They WILL test:**
- Slices vs arrays (length, capacity, append behavior)
- Interface implementation (implicit, duck typing)
- Error handling with `errors.New`, `fmt.Errorf`, `errors.Is`, `errors.As`
- Pointers vs value receivers on methods
- Defer, panic, recover — order of execution
- Maps — zero values, nil maps, concurrent access issues
- Structs — embedding vs inheritance
- `init()` function behavior

**They will NOT test at this band:**
- Reflection (`reflect` package internals)
- Unsafe pointer arithmetic
- Compiler optimizations or escape analysis
- Runtime internals (GC pause tuning, GOMAXPROCS fine-tuning)

**They WILL test:**
- Arrays and strings (two pointers, sliding window, frequency maps)
- Basic recursion and iteration

- BFS/DFS on simple trees or graphs

- Hash maps for O(1) lookup problems

**They will NOT test at this band:**
- Segment trees, Fenwick trees

## Go Fundamentals Checklist

Use this as your daily revision checklist. Mark each item as you feel confident.

### Basic Syntax

- [ ] **Variable declaration** — `var x int`, `:=` shorthand, multiple assignment
  - *Interview Q:* What is the difference between `var x int` and `x := 0`?

- [ ] **Constants** — `const`, `iota` for enumerated constants
  - *Interview Q:* How would you define HTTP method constants using `iota`?

- [ ] **Zero values** — every type has a zero value; `int` is 0, `string` is `""`, pointer is `nil`
  - *Interview Q:* What happens when you access a key that doesn't exist in a map?

- [ ] **Type conversion** — explicit only (`int(x)`, `string(b)` for runes), no implicit casting
  - *Interview Q:* Why does `string(65)` return `"A"` not `"65"`?

- [ ] **Multiple return values** — idiomatic error handling via `(result, error)` pairs
  - *Interview Q:* How do you ignore one of the return values in Go?

---

### Types

- [ ] **Arrays** — fixed size, value type (copied on assignment)
  - *Interview Q:* If you pass an array to a function and modify it, does the original change?

- [ ] **Slices** — reference type, has length and capacity, backed by array
  - *Interview Q:* What is the output of `len` and `cap` after `append` beyond capacity?

- [ ] **Slice internals** — header = (pointer, len, cap); `append` may allocate a new backing array
  - *Interview Q:* Why can modifying a sub-slice affect the original slice?

- [ ] **Maps** — unordered key-value, `make(map[K]V)`, nil map panics on write
  - *Interview Q:* What happens if you read from a nil map? What about write?

- [ ] **Strings** — immutable byte sequences, `range` iterates runes not bytes
  - *Interview Q:* How do you iterate over a string character by character in Go?

- [ ] **Runes** — `rune` is alias for `int32`, represents a Unicode code point
  - *Interview Q:* What is the difference between a byte and a rune?

- [ ] **Structs** — value types, can embed other structs for composition
  - *Interview Q:* How does struct embedding differ from inheritance in OOP?

- [ ] **Pointers** — `&variable` takes address, `*pointer` dereferences; no pointer arithmetic
  - *Interview Q:* When should you use a pointer receiver vs a value receiver?

---

### Functions

- [ ] **First-class functions** — functions are values, can be assigned to variables, passed as arguments
  - *Interview Q:* Write a function that takes another function as a parameter.

- [ ] **Closures** — inner function captures variables from outer scope
  - *Interview Q:* What is a common bug with closures inside goroutines in a loop?

- [ ] **Variadic functions** — `func sum(nums ...int)`, pass slice with `slice...`
  - *Interview Q:* How do you pass a slice to a variadic function?

- [ ] **Defer** — executes when surrounding function returns, LIFO order for multiple defers
  - *Interview Q:* What is the output order when three defer statements are used?

- [ ] **Named return values** — can name return variables, allows bare `return`
  - *Interview Q:* What is a potential pitfall of using named return values with defer?

---

### Interfaces

- [ ] **Implicit implementation** — no `implements` keyword; if a type has the methods, it satisfies the interface
  - *Interview Q:* How do you check if a value implements an interface at runtime?

- [ ] **Empty interface** — `interface{}` (or `any` in Go 1.18+) accepts any type
  - *Interview Q:* When would you use `interface{}` and what are the risks?

- [ ] **Type assertion** — `v, ok := i.(T)` to extract concrete type
  - *Interview Q:* What happens if you do a type assertion without the ok check and it fails?

- [ ] **Type switch** — `switch v := i.(type)` to handle multiple concrete types
  - *Interview Q:* Write a function that prints the type and value of any argument.

- [ ] **Stringer interface** — `fmt.Stringer` with `String() string` for custom printing
  - *Interview Q:* How do you make your struct print a custom format with `fmt.Println`?

- [ ] **io.Reader / io.Writer** — core interfaces for I/O composition
  - *Interview Q:* Why is `io.Reader` considered one of Go's most important interfaces?

---

### Error Handling

- [ ] **errors.New** — creates a simple error value
  - *Interview Q:* What is the difference between `errors.New` and `fmt.Errorf`?

- [ ] **fmt.Errorf with %w** — wraps an error to preserve the chain
  - *Interview Q:* How do you add context to an error without losing the original?

- [ ] **errors.Is** — checks if an error in the chain matches a target
  - *Interview Q:* When would `errors.Is` return true even for a wrapped error?

- [ ] **errors.As** — extracts a specific error type from the chain
  - *Interview Q:* Write code that extracts a custom error type from a wrapped error.

- [ ] **Custom error types** — struct implementing `error` interface with context fields
  - *Interview Q:* Why would you define a custom error type instead of using `errors.New`?

- [ ] **Panic and recover** — panic unwinds the stack; recover only works inside defer
  - *Interview Q:* When is it appropriate to use panic in production Go code?

## Concurrency Basics You MUST Know

### Topic 1 — Goroutines

**What to know:** A goroutine is a lightweight thread managed by the Go runtime. Start with `go func()`. Goroutines are cheap (2KB initial stack). The program exits when `main` returns, killing all goroutines.

**What they'll ask:** "What happens if main returns while goroutines are still running?"

**Code example:**
```go
package main

import (
    "fmt"
    "time"
)

func greet(name string) {
    fmt.Printf("Hello, %s\n", name)
}

func main() {
    go greet("Alice")
    go greet("Bob")
    // Without this sleep, main may exit before goroutines run
    time.Sleep(10 * time.Millisecond)
}
```

**Common mistake:** Relying on `time.Sleep` to wait for goroutines. Always use `sync.WaitGroup` or channels instead.

---

### Topic 2 — sync.WaitGroup

**What to know:** Used to wait for a collection of goroutines to finish. `Add(n)` increments counter, `Done()` decrements, `Wait()` blocks until counter hits zero.

**What they'll ask:** "How do you wait for 5 goroutines to complete?"

**Code example:**
```go
package main

import (
    "fmt"
    "sync"
)

func worker(id int, wg *sync.WaitGroup) {
    defer wg.Done()
    fmt.Printf("Worker %d done\n", id)
}

func main() {
    var wg sync.WaitGroup
    for i := 1; i <= 5; i++ {
        wg.Add(1)
        go worker(i, &wg)
    }
    wg.Wait()
    fmt.Println("All workers done")
}
```

**Common mistake:** Calling `wg.Add(1)` inside the goroutine instead of before launching it. This creates a race condition.

---

### Topic 3 — Channels (Unbuffered)

**What to know:** Unbuffered channel synchronizes sender and receiver. Send blocks until receiver is ready. Receive blocks until sender sends. Think of it as a rendezvous point.

**What they'll ask:** "Explain the difference between buffered and unbuffered channels."

**Code example:**
```go
package main

import "fmt"

func sum(nums []int, result chan int) {
    total := 0
    for _, n := range nums {
        total += n
    }
    result <- total
}

func main() {
    nums := []int{1, 2, 3, 4, 5}
    ch := make(chan int) // unbuffered
    go sum(nums, ch)
    total := <-ch
    fmt.Println("Sum:", total)
}
```

**Common mistake:** Sending to a nil channel blocks forever; receiving from a nil channel blocks forever. Always initialize channels with `make`.

---

### Topic 4 — Channels (Buffered)

**What to know:** Buffered channel has a capacity. Send blocks only when buffer is full. Receive blocks only when buffer is empty. Useful for decoupling producer/consumer speed.

**What they'll ask:** "When would you choose a buffered channel over an unbuffered one?"

**Code example:**
```go
package main

import "fmt"

func main() {
    ch := make(chan string, 3) // buffer of 3

    ch <- "first"
    ch <- "second"
    ch <- "third"
    // These sends don't block because buffer is not full

    fmt.Println(<-ch) // "first"
    fmt.Println(<-ch) // "second"
    fmt.Println(<-ch) // "third"
}
```

**Common mistake:** Assuming a buffered channel prevents all blocking. If all buffer slots are full and no one is reading, the sender will still block.

---

### Topic 5 — Select Statement

**What to know:** `select` waits on multiple channel operations simultaneously. It picks whichever channel is ready. If multiple are ready, it picks one at random. `default` makes it non-blocking.

**What they'll ask:** "How do you implement a timeout for a channel operation?"

**Code example:**
```go
package main

import (
    "fmt"
    "time"
)

func fetchData(ch chan string) {
    time.Sleep(2 * time.Second)
    ch <- "data arrived"
}

func main() {
    ch := make(chan string, 1)
    go fetchData(ch)

    select {
    case result := <-ch:
        fmt.Println("Got:", result)
    case <-time.After(1 * time.Second):
        fmt.Println("Timeout: operation took too long")
    }
}
```

**Common mistake:** Forgetting the `default` case when you want non-blocking behavior. Without `default`, `select` with no ready channels will block forever.

---

### Topic 6 — sync.Mutex

**What to know:** Mutex (mutual exclusion lock) protects shared data from concurrent access. `Lock()` acquires, `Unlock()` releases. Always defer `Unlock()` to prevent deadlocks.

**What they'll ask:** "How do you make a counter safe for concurrent access?"

**Code example:**
```go
package main

import (
    "fmt"
    "sync"
)

type SafeCounter struct {
    mu    sync.Mutex
    count int
}

func (c *SafeCounter) Increment() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.count++
}

func (c *SafeCounter) Value() int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.count
}

func main() {
    counter := &SafeCounter{}
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter.Increment()
        }()
    }
    wg.Wait()
    fmt.Println("Final count:", counter.Value()) // Always 1000
}
```

**Common mistake:** Using a value receiver on methods that call `Lock()`. The mutex gets copied and locking is ineffective. Always use pointer receivers with mutex.

---

### Topic 7 — Closing Channels and Range

**What to know:** `close(ch)` signals no more values will be sent. Receivers can range over a closed channel; the loop ends when the channel is drained and closed. Sending to a closed channel panics.

**What they'll ask:** "How does a receiver know when a channel is closed?"

**Code example:**
```go
package main

import "fmt"

func producer(ch chan int) {
    for i := 1; i <= 5; i++ {
        ch <- i
    }
    close(ch) // signal: no more values
}

func main() {
    ch := make(chan int, 5)
    go producer(ch)

    for val := range ch { // exits when channel closed and drained
        fmt.Println("Received:", val)
    }
    fmt.Println("Channel closed, done")
}
```

**Common mistake:** Closing a channel from the receiver side, or closing a channel multiple times. Only the sender should close a channel. Closing twice panics.

---

### Topic 8 — context.Context

**What to know:** `context.Context` carries deadlines, cancellation signals, and request-scoped values across API boundaries. Use `context.WithCancel`, `context.WithTimeout`, `context.WithDeadline`.

**What they'll ask:** "How do you cancel a goroutine using context?"

**Code example:**
```go
package main

import (
    "context"
    "fmt"
    "time"
)

func longTask(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            fmt.Println("Task cancelled:", ctx.Err())
            return
        default:
            fmt.Println("Working...")
            time.Sleep(500 * time.Millisecond)
        }
    }
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()
    go longTask(ctx)
    time.Sleep(3 * time.Second)
}
```

**Common mistake:** Not calling `cancel()`. Always `defer cancel()` immediately after creating a context with `WithCancel` or `WithTimeout` to free resources.

---

### Topic 9 — Race Conditions and -race Flag

**What to know:** A race condition occurs when two goroutines access shared memory concurrently and at least one is writing. Go's race detector (`go run -race` or `go test -race`) detects these at runtime.

**What they'll ask:** "How do you detect race conditions in Go?"

**Code example:**
```go
// BAD: Race condition
package main

import (
    "fmt"
    "sync"
)

func main() {
    counter := 0
    var wg sync.WaitGroup
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter++ // DATA RACE: concurrent read-modify-write
        }()
    }
    wg.Wait()
    fmt.Println(counter) // non-deterministic result
}
// Fix: use sync.Mutex or sync/atomic
```

**Common mistake:** Thinking that incrementing a simple integer is safe because it "looks atomic." At the CPU level, `counter++` is a read-modify-write, which is three operations.

---

### Topic 10 — Goroutine Leaks

**What to know:** A goroutine leak happens when a goroutine is started but never terminates, consuming memory indefinitely. Common causes: blocking forever on a channel receive with no sender, or no cancellation path.

**What they'll ask:** "What causes goroutine leaks and how do you prevent them?"

**Code example:**
```go
// BAD: goroutine leak
func leaky() {
    ch := make(chan int)
    go func() {
        val := <-ch // blocks forever, no one sends to ch
        fmt.Println(val)
    }()
    // ch goes out of scope; goroutine is stuck
}

// GOOD: use context for cancellation
func notLeaky(ctx context.Context) {
    ch := make(chan int)
    go func() {
        select {
        case val := <-ch:
            fmt.Println(val)
        case <-ctx.Done():
            return // exits cleanly
        }
    }()
}
```

**Common mistake:** Not providing a cancellation mechanism (via context or a done channel) for long-running goroutines. In HTTP handlers, use the request context.

### Go-Specific DSA Tips

```go
// Use sort.Slice for custom sorting
sort.Slice(arr, func(i, j int) bool {
    return arr[i] < arr[j]
})

// Use a map as a set
seen := make(map[int]bool)
seen[val] = true
if seen[val] { /* already present */ }

// Append-based stack
stack := []int{}
stack = append(stack, val)        // push
top := stack[len(stack)-1]        // peek
stack = stack[:len(stack)-1]      // pop

// Append-based queue (use container/list for O(1) dequeue)
queue := []int{}
queue = append(queue, val)         // enqueue
front := queue[0]                  // peek front
queue = queue[1:]                  // dequeue (O(n), ok for interviews)
```

## System Design Basics

At the 8–15 LPA band, system design rounds are **introductory**. You are not expected to design Twitter from scratch. You need to show structured thinking: clarify requirements, estimate scale, propose data model, sketch components, mention trade-offs.

**Framework for every system design answer:**
1. Clarify requirements (2 min)
2. Estimate scale (1 min)
3. High-level architecture (5 min)
4. Data model (3 min)
5. Key trade-offs (2 min)

### Design 1 — URL Shortener

**Clarify:**
- Expected QPS? (read-heavy: ~10k reads/100 writes per second)
- URL expiry? Custom short codes?
- Analytics needed?

**High-level:**
```
Client -> Load Balancer -> Go API Servers -> Redis (cache) -> Postgres (storage)
                                          -> ID Generator (distributed)
```

**Data model:**
```sql
TABLE urls (
    id          BIGINT PRIMARY KEY,
    short_code  VARCHAR(8) UNIQUE,
    long_url    TEXT,
    created_at  TIMESTAMP,
    expires_at  TIMESTAMP
)
```

**Key decisions:**
- Short code generation: Base62 encoding of an auto-increment ID or random 6-char code
- Cache: Redis with TTL, cache the top 20% of URLs that get 80% of traffic
- Redirect: HTTP 301 (browser caches, fewer hits) vs 302 (always hits your server, better analytics)

**Trade-offs to mention:**
- 301 reduces server load but makes analytics inaccurate
- Hash collisions with random codes require retry logic
- Distributed ID generation: use a counter per region or a service like Snowflake

### Design 2 — Simple Rate Limiter

**Clarify:**
- Per user or per IP? Sliding window or fixed window?
- What happens when limit exceeded — drop or queue?
- Distributed (multiple servers) or single node?

**Algorithms to know:**
- **Fixed Window:** Counter resets every N seconds. Simple, but allows burst at window boundary.
- **Sliding Window Log:** Store timestamps of requests; count in last N seconds. Memory-heavy.
- **Token Bucket:** Tokens refill at rate R, each request consumes one token. Smooth bursts.

**Go implementation sketch (Token Bucket with Redis):**
```

EXPIRE key 60  (set TTL on first increment)
If count > limit -> reject request
```

**Data model:**
```

  value: 47 (requests in current window)
  TTL:   13 (seconds remaining in window)
```

**Trade-offs:**
- Redis INCR + EXPIRE is atomic but race condition exists between the two commands; use Lua script or SETNX for atomicity
- In-memory rate limiter is faster but breaks with multiple pods

### Design 3 — Basic Cache (LRU Cache)

**Clarify:**
- Max capacity? Eviction policy (LRU, LFU, TTL)?
- Thread-safe needed?

**LRU Cache design:**
- Data structure: HashMap + Doubly Linked List
- HashMap gives O(1) lookup by key
- Doubly Linked List maintains recency order
- On access: move node to head
- On eviction: remove tail node

**Go approach:**
```
type LRUCache struct {
    capacity int
    cache    map[int]*Node
    head     *Node  // most recently used
    tail     *Node  // least recently used
}
```

**Trade-offs:**
- For distributed cache: use Redis with `maxmemory-policy allkeys-lru`
- Write-through vs write-back: write-through is safer (writes to DB immediately), write-back is faster but risks data loss
- Cache invalidation: TTL-based, event-driven (pub/sub), or explicit delete on write

### Pattern 1 — Reverse a String

```go
package main

import "fmt"

func reverseString(s string) string {
    runes := []rune(s) // handle unicode correctly
    left, right := 0, len(runes)-1
    for left < right {
        runes[left], runes[right] = runes[right], runes[left]
        left++
        right--
    }
    return string(runes)
}

func main() {
    fmt.Println(reverseString("hello"))   // "olleh"
    fmt.Println(reverseString("golang"))  // "gnalog"
    fmt.Println(reverseString("नमस्ते"))  // handles Unicode
}
```

**Why `[]rune` not `[]byte`:** A byte is 8 bits; a rune is a Unicode code point (up to 32 bits). Multi-byte characters (like Hindi, Chinese, emoji) will be corrupted if you reverse bytes.

### Pattern 7 — Fibonacci (Iterative and Recursive)

```go
package main

import "fmt"

// Iterative: O(n) time, O(1) space
func fibIterative(n int) int {
    if n <= 1 {
        return n
    }
    prev, curr := 0, 1
    for i := 2; i <= n; i++ {
        prev, curr = curr, prev+curr
    }
    return curr
}

// Memoized recursive: O(n) time, O(n) space
func fibMemo(n int, memo map[int]int) int {
    if n <= 1 {
        return n
    }
    if val, ok := memo[n]; ok {
        return val
    }
    memo[n] = fibMemo(n-1, memo) + fibMemo(n-2, memo)
    return memo[n]
}

func main() {
    fmt.Println(fibIterative(10)) // 55
    memo := make(map[int]int)
    fmt.Println(fibMemo(10, memo)) // 55
}
```

### Pattern 8 — Check if a String is a Palindrome

```go
    "fmt"
    "strings"
    "unicode"

// Simple palindrome check
func isPalindrome(s string) bool {
    runes := []rune(strings.ToLower(s))
    left, right := 0, len(runes)-1
    for left < right {
        // skip non-alphanumeric (for interview variant)
        for left < right && !unicode.IsLetter(runes[left]) && !unicode.IsDigit(runes[left]) {
            left++
        }

            right--
        }
        if runes[left] != runes[right] {
            return false
        }
        left++
        right--
    }
    return true
}

    fmt.Println(isPalindrome("A man a plan a canal Panama")) // true
    fmt.Println(isPalindrome("hello"))           // false
}
```

### Pattern 10 — Binary Search

```go
package main

import "fmt"

func binarySearch(arr []int, target int) int {
    left, right := 0, len(arr)-1
    for left <= right {
        mid := left + (right-left)/2 // avoids integer overflow
        if arr[mid] == target {
            return mid
        } else if arr[mid] < target {
            left = mid + 1
        } else {
            right = mid - 1
        }
    }
    return -1
}
```

### Pattern 11 — Stack Implementation

```go
    "errors"
    "fmt"

    items []T

    s.items = append(s.items, item)
}

func (s *Stack[T]) Pop() (T, error) {
    var zero T
    if len(s.items) == 0 {
        return zero, errors.New("stack is empty")
    }
    top := s.items[len(s.items)-1]
    s.items = s.items[:len(s.items)-1]
    return top, nil
}

    var zero T
    if len(s.items) == 0 {
        return zero, errors.New("stack is empty")
    }
    return s.items[len(s.items)-1], nil
}

func (s *Stack[T]) IsEmpty() bool {
    return len(s.items) == 0
}

func main() {
    s := &Stack[int]{}
    s.Push(1)
    s.Push(2)
    s.Push(3)
    val, _ := s.Pop()
    fmt.Println(val) // 3
}
```

### Pattern 12 — Valid Parentheses

```go
    stack := []rune{}

        ')': '(',
        ']': '[',
        '}': '{',

    for _, ch := range s {

            stack = stack[:len(stack)-1]
        }
    }
    return len(stack) == 0
}

    fmt.Println(isValid("([)]"))    // false
```

### Pattern 16 — Sliding Window (Max Sum of K Elements)

```go
package main

import "fmt"

func maxSumSubarray(nums []int, k int) int {
    if len(nums) < k {
        return 0
    }
    windowSum := 0
    for i := 0; i < k; i++ {
        windowSum += nums[i]
    }
    maxSum := windowSum
    for i := k; i < len(nums); i++ {
        windowSum += nums[i] - nums[i-k] // slide: add new, remove old
        if windowSum > maxSum {
            maxSum = windowSum
        }
    }
    return maxSum
}

func main() {
    nums := []int{2, 1, 5, 1, 3, 2}
    fmt.Println(maxSumSubarray(nums, 3)) // 9 (5+1+3)
```

### Pattern 17 — BFS on a Graph (Adjacency List)

```go
package main

import "fmt"

func bfs(graph map[int][]int, start int) []int {
    visited := make(map[int]bool)
    queue := []int{start}
    visited[start] = true
    order := []int{}

    for len(queue) > 0 {
        node := queue[0]
        queue = queue[1:]
        order = append(order, node)

        for _, neighbor := range graph[node] {
            if !visited[neighbor] {
                visited[neighbor] = true
                queue = append(queue, neighbor)
            }
        }
    }
    return order
}

        3: {6},
        4: {},
        5: {},
        6: {},

}
```

### Pattern 19 — Channel Fan-Out / Fan-In

```go
package main

import (
    "fmt"
    "sync"
)

// Fan-out: distribute work to multiple workers
func fanOut(input <-chan int, numWorkers int) []<-chan int {
    channels := make([]<-chan int, numWorkers)
    for i := 0; i < numWorkers; i++ {
        ch := make(chan int, 10)
        channels[i] = ch
        go func(out chan<- int) {
            for val := range input {
                out <- val * val // square the number
            }
            close(out)
        }(ch)
    }
    return channels
}

// Fan-in: merge multiple channels into one
func fanIn(channels []<-chan int) <-chan int {
    merged := make(chan int, 10)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for val := range c {
                merged <- val
            }
        }(ch)
    }
    go func() {
        wg.Wait()
        close(merged)
    }()
    return merged
}

func main() {
    input := make(chan int, 5)
    for i := 1; i <= 5; i++ {
        input <- i
    }
    close(input)

    workers := fanOut(input, 3)
    results := fanIn(workers)
    for r := range results {
        fmt.Println(r)
    }
}
```

### Pattern 20 — Simple Middleware in net/http

```go
package main

import (
    "fmt"
    "log"
    "net/http"
    "time"
)

// Middleware type: wraps an http.Handler
type Middleware func(http.Handler) http.Handler

// Logging middleware
func LoggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}

// Auth middleware
func AuthMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        token := r.Header.Get("Authorization")
        if token == "" {
            http.Error(w, "Unauthorized", http.StatusUnauthorized)
            return
        }
        next.ServeHTTP(w, r)
    })
}

func helloHandler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "Hello, World!")
}

func chain(h http.Handler, middlewares ...Middleware) http.Handler {
    for i := len(middlewares) - 1; i >= 0; i-- {
        h = middlewares[i](h)
    }
    return h
}

func main() {
    handler := chain(
        http.HandlerFunc(helloHandler),
        LoggingMiddleware,
        AuthMiddleware,
    )
    http.Handle("/", handler)
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

## Sample Interview Q&A

### Q1: What is the difference between a goroutine and an OS thread?

**Answer:** An OS thread is managed by the operating system and has a fixed stack size (typically 1-8 MB). A goroutine is managed by the Go runtime, starts with a small stack (2 KB), and grows as needed. Goroutines are multiplexed onto a smaller number of OS threads by the Go scheduler. You can run millions of goroutines on a single machine because of their lightweight nature, whereas the same is impractical with OS threads due to memory overhead and context-switch cost.

---

### Q2: Explain the difference between make and new in Go.

**Answer:** `new(T)` allocates zeroed storage for type T and returns a pointer `*T`. It is used for value types when you need a pointer. `make(T, ...)` is used only for slices, maps, and channels. It allocates and initializes the internal data structure and returns the initialized (not zeroed) value of type T, not a pointer. For example, `make([]int, 5)` returns a slice header pointing to a zeroed array of 5 ints. Using `new([]int)` would return a pointer to a nil slice header, which is rarely what you want.

---

### Q3: What is a defer statement and in what order do multiple defers execute?

**Answer:** A `defer` statement schedules a function call to execute when the surrounding function returns, regardless of whether it returns normally or panics. Multiple defer statements execute in LIFO (last-in, first-out) order — the last defer declared runs first. This is useful for cleanup: open a resource, defer its close. Defer arguments are evaluated immediately when the defer statement is encountered, not when it executes. This matters for things like `defer fmt.Println(x)` where `x` is captured by value at the point of the defer call.

---

### Q4: How does Go handle errors? Why doesn't Go have exceptions?

**Answer:** Go uses explicit error values rather than exceptions. Functions that can fail return an `error` as their last return value. Callers check `if err != nil`. The design philosophy is that error handling should be explicit and visible in the code flow, making it harder to accidentally ignore errors (unlike exceptions which can be silently uncaught). Go 1.13 added `%w` in `fmt.Errorf` to wrap errors, and `errors.Is`/`errors.As` to inspect the error chain. This approach makes error propagation transparent but can be verbose for deeply nested calls.

---

### Q5: What is the difference between a nil interface and an interface holding a nil value?

**Answer:** An interface has two components: a type and a value. A nil interface has both set to nil — it is entirely empty. An interface holding a nil value has a concrete type set but its value pointer is nil. The key gotcha is: `var err *MyError = nil; var i error = err` results in `i != nil` being true because the interface has a non-nil type. This is a common source of bugs when returning concrete nil pointer types where the caller checks `if err != nil`. Always return the `error` interface directly, never a typed nil pointer.

---

### Q6: How do you prevent a goroutine leak?

**Answer:** Goroutine leaks occur when a goroutine blocks indefinitely — typically waiting on a channel with no sender, or a channel that will never be closed. Prevention strategies: (1) Use `context.Context` for cancellation; pass context through call chains and select on `ctx.Done()` in goroutines. (2) Always ensure channels will eventually be closed or received from. (3) Use `time.After` or `context.WithTimeout` for operations that may hang. (4) Use tools like `goleak` (Uber's library) in tests to detect leaked goroutines. At the HTTP handler level, the request context is automatically cancelled when the client disconnects.

---

### Q7: What is the difference between a value receiver and pointer receiver?

**Answer:** A value receiver gets a copy of the struct; modifications do not affect the original. A pointer receiver gets a reference to the original struct; modifications persist. Use pointer receivers when: the method needs to modify the receiver's fields, the struct is large (avoid copying), or you need to maintain consistency (if any method uses a pointer receiver, all methods should, so the type implements both value and pointer interface sets). Value receivers are appropriate for small, immutable types or when intentional copying is desired. Pointer receiver methods can be called on addressable values automatically by Go.

---

### Q8: What is an interface in Go, and how is it different from Java interfaces?

**Answer:** In Go, interfaces are satisfied implicitly. If a type has all the methods defined in an interface, it automatically implements that interface — no `implements` keyword needed. This is called structural typing (or duck typing). In Java, a class must explicitly declare `implements MyInterface`. The Go approach decouples the interface definition from the type definition, allowing you to define interfaces in consuming packages (accept what you need, not what the type offers). This makes code more composable and testable. Go interfaces are typically small (1-3 methods), following the principle that the larger the interface, the weaker the abstraction.

---

### Q9: How does Go's garbage collector work (at a high level)?

**Answer:** Go uses a concurrent, tri-color mark-and-sweep garbage collector. The three colors are: white (not yet reached), grey (reached but children not yet scanned), and black (fully scanned). The GC starts by marking all root objects grey, then processes grey objects (marking their children grey, themselves black), until no grey objects remain. White objects at the end are garbage and are swept. The key feature is that it runs concurrently with the program to minimize stop-the-world pauses (typically under 1ms in modern Go). The GC is triggered when the heap grows to a certain size ratio, controlled by the `GOGC` environment variable (default 100, meaning GC triggers when heap doubles).

---

### Q10: What is the difference between sync.Mutex and sync.RWMutex?

**Answer:** `sync.Mutex` has two operations: `Lock()` and `Unlock()`. Only one goroutine can hold the lock at a time, whether reading or writing. `sync.RWMutex` supports multiple concurrent readers: `RLock()`/`RUnlock()` for readers (multiple can hold simultaneously) and `Lock()`/`Unlock()` for writers (exclusive). Use `RWMutex` when your data has a high read-to-write ratio — it improves throughput by allowing concurrent reads. Important: a pending writer in `RWMutex` blocks new readers from acquiring the lock (to prevent writer starvation), so a read-heavy workload with occasional long reads can still cause writer delays.

---

### Q11: What happens when you close a channel in Go?

**Answer:** Closing a channel signals that no more values will be sent on it. Receivers can still read remaining values from the buffer. Once the buffer is drained, any further receive operations return the zero value of the channel's type along with `false` as the ok flag (`val, ok := <-ch`). Ranging over a closed channel terminates the loop after the buffer is drained. Rules: only the sender should close a channel (closing from the receiver is a design smell); closing a nil channel panics; closing an already-closed channel panics; sending to a closed channel panics. Closing is primarily a signaling mechanism, not a cleanup operation.

---

### Q12: What is the `init` function in Go?

**Answer:** The `init` function is a special function that runs automatically before `main`. Each package can have multiple `init` functions across multiple files. They run in the order the files are presented to the compiler, after all variable initializations in the package are done. You cannot call `init` explicitly or take its address. Common uses: registering drivers (database/sql drivers register themselves in `init`), setting up global state, validating configuration. Be careful with `init` — it can make code harder to test and reason about due to hidden side effects. Prefer explicit initialization functions called from `main` when possible.

---

### Q13: How do slices share underlying arrays? Show with an example.

**Answer:** A slice is a three-field struct: pointer to an underlying array, length, and capacity. When you slice a slice (`b := a[1:3]`), `b` shares the same backing array as `a`. Modifying elements of `b` also modifies `a`. However, after `append` causes a new backing array to be allocated (when length would exceed capacity), the slices diverge.

```go
a := []int{1, 2, 3, 4, 5}
b := a[1:3]    // b = [2, 3], shares a's array
b[0] = 99      // modifies a[1] too
fmt.Println(a) // [1 99 3 4 5]

b = append(b, 100, 200, 300, 400) // exceeds a's capacity
b[0] = 77      // now b has its own array, a is unaffected
fmt.Println(a) // [1 99 3 4 5] — unchanged
```

---

### Q14: What are channels in Go and when would you use them?

**Answer:** Channels are typed conduits for communication between goroutines. They enforce synchronization: sending to an unbuffered channel blocks until a receiver is ready, and receiving blocks until a sender sends. Use channels when: (1) one goroutine produces data and another consumes it (pipeline), (2) you need to signal completion or cancellation (`done chan struct{}`), (3) you want to distribute work across workers (fan-out), (4) you need to collect results from multiple goroutines (fan-in). The Go mantra is "do not communicate by sharing memory; instead, share memory by communicating." However, not everything needs channels — for protecting a shared value, a `sync.Mutex` is simpler and clearer.

---

### Q15: What is context in Go and why is it important for HTTP services?

**Answer:** `context.Context` carries deadlines, cancellation signals, and request-scoped values across API and goroutine boundaries. In HTTP services, every incoming request gets a context from the request (`r.Context()`). This context is automatically cancelled when the client disconnects or the server's timeout fires. If you pass this context to all downstream operations (database queries, HTTP calls to other services, goroutines), all of them can react to the cancellation and stop work immediately instead of wasting resources. This is critical for a service's health: without context propagation, a cancelled client request continues consuming CPU and database connections all the way down the call chain.

---

### Q16: Explain Go modules and what `go.mod` contains.

**Answer:** Go modules are the dependency management system introduced in Go 1.11 and made the default in Go 1.16. A module is a collection of Go packages with a `go.mod` file at its root. `go.mod` contains: the module path (the import path prefix for all packages in the module), the minimum Go version required, and the `require` block listing direct and indirect dependencies with their minimum version. Go modules use Minimum Version Selection (MVS): when multiple versions of a dependency are required, Go picks the minimum version that satisfies all requirements — this is deterministic and reproducible without a separate lock file. The `go.sum` file records cryptographic hashes of module content for security verification.

### Q17: How do you write a unit test in Go?

**Answer:** Go has built-in testing via the `testing` package. Test files end in `_test.go`. Test functions start with `Test` and take `*testing.T`. Run with `go test ./...`.

```go
// calculator.go
func Add(a, b int) int { return a + b }

// calculator_test.go
package main

import "testing"

func TestAdd(t *testing.T) {
    tests := []struct {
        a, b, expected int
    }{
        {1, 2, 3},
        {-1, 1, 0},
        {0, 0, 0},
    }
    for _, tt := range tests {
        result := Add(tt.a, tt.b)
        if result != tt.expected {
            t.Errorf("Add(%d, %d) = %d; want %d", tt.a, tt.b, result, tt.expected)
        }
    }
}
```

Table-driven tests are the idiomatic Go testing pattern — they consolidate multiple cases and make adding new test cases easy.

### Q18: What is the difference between `fmt.Println` and `log.Println`?

**Answer:** `fmt.Println` writes to standard output (stdout) with no timestamp or prefix. `log.Println` writes to standard error (stderr) by default and prepends a timestamp and optional prefix. `log.Fatal` is `log.Println` followed by `os.Exit(1)`. `log.Panic` is `log.Println` followed by `panic`. In production code, use a structured logging library (like `log/slog` in Go 1.21+, or `zap`, `zerolog`) which outputs JSON, supports log levels, and is more suitable for log aggregation systems. `fmt.Println` is appropriate for CLI tools and quick debugging; `log` is for application logging; structured logging libraries are for production services.

### Q19: How would you handle a panic gracefully in a web server?

**Answer:** In a web server, an unhandled panic crashes the entire process, taking down all concurrent requests. Use a recovery middleware to catch panics per request and return a 500 response instead. This is implemented using `defer` and `recover()`:

```go
func RecoveryMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Printf("panic recovered: %v\n%s", err, debug.Stack())
                http.Error(w, "Internal Server Error", http.StatusInternalServerError)
            }
        }()
        next.ServeHTTP(w, r)
    })
}
```

`recover()` only stops the panic in the current goroutine. Panics in other goroutines (spawned from a handler) must have their own recovery. Gin and other frameworks include this middleware by default.

---

### Q20: What is the difference between concurrency and parallelism in Go?

**Answer:** Concurrency is about dealing with multiple tasks that can start, run, and complete in overlapping time periods — it is a program structure. Parallelism is about actually running multiple computations simultaneously, which requires multiple CPUs. In Go, goroutines provide concurrency. Whether they run in parallel depends on `GOMAXPROCS` (default: number of CPU cores). You can have a concurrent program running on one CPU core (goroutines take turns) or a parallel program running on multiple cores. Rob Pike's summary: "Concurrency is about dealing with lots of things at once. Parallelism is about doing lots of things at once." Concurrency is the design; parallelism is execution. Go excels at both.

```go
package main

import (
    "fmt"
    "sort"
)

func findAllPairs(nums []int, target int) [][2]int {
    // map from value to list of indices where it appears
    indexMap := make(map[int][]int)
    var result [][2]int

    for i, num := range nums {
        complement := target - num
        if indices, found := indexMap[complement]; found {
            for _, j := range indices {
                if j < i {
                    result = append(result, [2]int{j, i})
                }
            }
        }
        indexMap[num] = append(indexMap[num], i)
    }

    sort.Slice(result, func(a, b int) bool {
        return result[a][0] < result[b][0]
    })
    return result
}

func main() {
    nums := []int{1, 2, 3, 2, 1}
    target := 4
    pairs := findAllPairs(nums, target)
    fmt.Println(pairs) // [[0 2] [1 3] [2 4]]
}
```

The time complexity is O(n) for the iteration plus O(k log k) for sorting the result, where k is the number of pairs found. Space is O(n) for the map.

## Quick Reference Cheat Sheet

### Common Gotchas at Interviews

```go
// 1. Loop variable capture in goroutines — WRONG
for i := 0; i < 3; i++ {
    go func() { fmt.Println(i) }() // prints 3,3,3 not 0,1,2
}

// CORRECT: pass as parameter
for i := 0; i < 3; i++ {
    go func(n int) { fmt.Println(n) }(i)
}

// 2. Nil map write — PANICS
var m map[string]int
m["key"] = 1 // panic: assignment to entry in nil map

// CORRECT
m := make(map[string]int)
m["key"] = 1

// 3. Interface nil trap
type MyErr struct{ msg string }
func (e *MyErr) Error() string { return e.msg }

func getError() error {
    var err *MyErr = nil
    return err // returns non-nil interface! (type is *MyErr, value is nil)
}
// CORRECT: return nil directly
func getError() error {
    return nil
}

// 4. Defer with loop — resource leak
for _, file := range files {
    f, _ := os.Open(file)
    defer f.Close() // BAD: defers don't run until function returns
    process(f)
}
// CORRECT: wrap in an anonymous function or helper
for _, file := range files {
    func() {
        f, _ := os.Open(file)
        defer f.Close()
        process(f)
    }()
}

// 5. Modifying slice while ranging
nums := []int{1, 2, 3, 4}
for i, v := range nums {
    if v == 2 {
        nums = append(nums, 99) // range uses a copy of the slice header
    }
    _ = i
}
```

| String concatenation with + | O(n²) | Use strings.Builder for loops |
| strings.Builder WriteString | O(1) amortized | Always prefer for building strings in loops |
